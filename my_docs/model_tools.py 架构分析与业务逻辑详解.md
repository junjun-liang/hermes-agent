# model_tools.py 架构分析与业务逻辑详解

## 1. 系统概述

`model_tools.py` 是 Hermes-Agent 的**工具编排层（Tool Orchestration Layer）**，作为工具注册表（`tools.registry`）的薄封装，负责：

- **工具发现与注册**：自动导入所有工具模块，触发自注册机制
- **Schema 生成与过滤**：根据启用的工具集生成 OpenAI 格式的工具定义
- **工具调用分发**：将 LLM 的函数调用路由到对应的工具处理器
- **异步桥接**：解决同步/异步执行环境的兼容性问题
- **参数类型强制转换**：将 LLM 输出的字符串参数转换为正确的 JSON Schema 类型

### 1.1 核心设计原则

| 设计原则 | 说明 |
|---------|------|
| **薄封装** | 不重复实现工具逻辑，仅做编排和路由 |
| **自注册** | 工具模块通过 `registry.register()` 自注册，无需手动维护列表 |
| **持久事件循环** | 避免 `asyncio.run()` 的创建-销毁循环，防止客户端绑定到死循环 |
| **线程安全** | 主线程和工作线程各自拥有独立的事件循环 |
| **参数类型安全** | 自动将字符串 "42" → 42，"true" → True |

**关键设计决策：**

1. **持久事件循环**：使用 `_tool_loop`（主线程）和 `_worker_thread_local.loop`（工作线程）存储长期存在的事件循环，避免 `asyncio.run()` 创建-销毁循环导致的客户端绑定问题

2. **线程隔离**：工作线程使用线程本地存储（`threading.local()`）保存各自的事件循环，避免与主线程竞争

3. **async 上下文检测**：如果在 async 上下文中调用（如 Gateway），则在新线程中运行 `asyncio.run()`，避免嵌套事件循环错误

**工具注册机制：**

每个工具模块在导入时会调用 `registry.register()` 自注册，例如：

```python
# tools/terminal_tool.py (示例)
from tools.registry import registry

registry.register(
    name="terminal",
    schema={...},           # OpenAI function schema
    handler=terminal_handler,  # 处理函数
    toolset="terminal",     # 所属工具集
    check_fn=check_terminal,  # 可用性检查函数
)
```

**关键逻辑说明：**

1. **工具集解析策略**：
   - `enabled_toolsets` 模式：仅包含指定工具集的工具
   - `disabled_toolsets` 模式：包含所有工具集，排除指定工具集
   - 默认模式：包含所有工具集

2. **Legacy 工具集支持**：支持旧版 `_tools` 后缀的工具集名称（如 `web_tools` → `web_search`, `web_extract`）

3. **动态 Schema 构建**：
   - `execute_code` 工具的 schema 动态构建，仅列出当前会话实际可用的沙箱工具
   - `browser_navigate` 的描述根据 `web_search`/`web_extract` 的可用性动态调整

4. **全局状态**：`_last_resolved_tool_names` 保存上次解析的工具名列表，供 `code_execution_tool` 使用
